Frontendの定義

  • Webサイト/Webアプリケーションの中のブラウザで動く部分
  • だいたいHTML/CSS/JavaScriptで作られている
  • JavaScriptでサーバからデータを取得して（Ajax）、HTMLの一部を書き換えたり
  • ユーザーのアクションをもとにJavaScriptが何かしたり

背景

サーバ側でHTMLを構築していて、ブラウザではそれを表示するだけだった時代

ブラウザの進化やHTML5周辺APIの誕生により、ブラウザでできることが増えた

UIの改善やモバイル環境への対応のために、フロントエンドで様々な処理をするようになった

よく使われる技術

  • gulp, webpack, CommonJS
  • ES6, ES7, browserify, babel
  • Sass, PostCSS
  • AWS, BaaS
  • GitHub, CI, Test

JSON Schema in Web Frontend

  • JSON Schema：JSONの構造をJSONで定義。IETFに仕様がある。
  • minimum,format,typeなどを定義できる。$refという参照も使える。validationに使う。
  • Hyper-Schema：RESTAPIのエンドポイントの定義に使ったりする。APIドキュメントの自動生成。HTTPクライアントの自動生成などができる。
  • 大体の言語で使える。Rubyでも使える。prmd GEM
  • スマホのリッチ広告をやっている社内ベンチャーでの導入事例
  • メールのやり取りベースで広告（のhtmlタグ）を作っていた。->取引先が増えると、自動化したくなった。
    AWS上で稼働するタグ発行ツールを作った。
  • サーバサイドはRESTfulAPI。golang。フロントはNode.js,BABEL,flow-type,React-Reduxなど
  • フロントとバックの連携のためのドキュメントとしてJSON Schemaを使う。
  • さらにJSON SchemaをベースにAPIのモックが自動生成できるので、バックエンドの実装を待たずにフロントが作れる。
  • JSON Schemaをもとに、JavaScriptでHTMLのformタグを自動生成とvalidationをする。
  • JSON Schema自体は手書きなので、ちょっと大変。prmdだとymlで書ける。
  • 更新時はバックとフロントで同時にリリースが必要なので、APIのバージョンとJSON Schemaのバージョンがsyncすると良さそう。
  • validationも両方でかけるべきだが、JSON Schemaをベースにすれば共通のルールになる。
